# Meguru — 昨夜の Wiki Absorb 失敗の原因特定・修正・約1ヶ月分リカバリ + last_synced 恒久対策 change

「昨夜失敗した Wiki Absorb をこのセッションで引き継いで解決・リカバリできる?」から始まり、原因特定 → 修正 → seed 後 1,541 件のリカバリ → last_synced 過去日の原因究明 → 恒久対策の OpenSpec change 起票・apply・archive・PR マージまで一気通貫。

## 障害: 昨夜の一括 Wiki Absorb が commit ゼロで見かけ上完走
- 約 1 ヶ月分（3/3〜7/10）・**79 shard / 6,509 エントリ**の一括 wiki absorb が「見かけ上完走」しつつ **記事を 1 件も書いていなかった**。
- 決定的証拠: `shards/**` 配下の全記事 mtime が **6/16（seed 日）のまま**。wiki リポは clean（`inbox/raw-entries/` は gitignore なので正常）。`pending_weekly_absorb.json` は Higashinippori6Chome の 1 shard だけ（大口 shard が timeout で唯一「明示失敗」記録、残り 78 shard は exit 0 無編集を orchestrator が「成功=no-op」と誤認）。

## 真因（2 バグ、`Plugins-Meguru/business-curator/scripts/tools/absorb_common.py`）
1. **`AbsorbExecutor` が `claude -p` を cwd 未指定で起動** → `wiki_catchup.sh` の `cd business-curator` を継承。absorb プロンプトが相対パスを **2 つの異なるルート**に混在（`skills/wiki/*-curator.md` は business-curator にのみ / `shards/…`・`inbox/raw-entries/…` は wiki にのみ存在）。結果 claude は curator ポリシーは読めても `shards/…` を解決できず**無編集で exit 0**。さらに `--permission-mode` 未指定で headless 編集も通らない。
2. （修正1適用後に露呈）**`--add-dir` 直後の positional prompt が「もう一つのディレクトリ」として食われ**、prompt 不在 → stdin 待ちで失敗 → **prompt を stdin 渡し**に変更。

## 修正（`Plugins-Meguru@717a23e`、テスト 27 green）
- cwd=wiki 起動 / prompt stdin 渡し / curator を絶対パス参照 + `--add-dir business-curator` / `--permission-mode acceptEdits` / commit の **パス限定 staging**（`git add -A` は並列時に他 shard を巻き込むため）。
- 検証: runbook の「nested `claude -p` はブロック」前提は今回当てはまらず、この環境で `claude -p` が動く（PONG/exit 0）→ **セッション内でリカバリ実行可能**と判明。実コードパス（`run_absorb`）で Nishioi6Chome 6 件を absorb し良質な narrative commit `22a8a73` を生成して修正の正しさを実証（ログ列挙でなく「竣工写真リンク不備→3日遅れ訂正」等の判断込み narrative 吸収）。
- **commit_shard の 2 次バグ**: パス限定 staging が `shards/{kind}/{slug}`（拡張子なし）でファイル型 shard `…{slug}.md` にマッチせず「clean」誤判定 → ファイル型/ディレクトリ型 両対応に修正。

## リカバリ（seed 後 = 6/16 以降の新規分のみ）
- 選択肢を実数提示（seed後 1,541件/64shard・全量 6,509件/79shard・失敗1件のみ179件）→ ユーザーが **seed 後のみ**を選択。
- nohup で detach、4 shard 並列、barrier 方式（全 absorb 完了後に直列 commit）。**63 shard / 1,535 件・失敗ゼロ**。MeguruWiki に 64 個の per-shard commit（`[catchup]`/`[catchup-recovery]`）。昨夜明示失敗した Higashinippori6Chome も回収（`eb3b8de`, 32 entries）。`pending_weekly_absorb.json` クリア + cursor を 2026-07-15 に更新。
- この修正で daily パイプライン Step 7（現在 `wiki_auto_absorb: false`）を有効化しても正しく動く土台ができた。

## last_synced 過去日の原因究明 → 恒久対策 change
- ユーザー質問「wiki の last sync 日付が過去日なのはなぜ?」。原因: 案件記事 frontmatter の **`jooto.last_synced`** は Jooto ボードのメタ取り込み時刻で、seed 時（4/14・6/16）に一度書かれたきり。メール absorb は本文しか触らず `jooto.*` frontmatter は更新経路が無い。→ **リカバリのバグではなく別系統フィールド**。本文の鮮度は最新化されている。
- ユーザーが「Jooto→wiki frontmatter 同期ステップ新設」を選択 → OpenSpec change **`add-jooto-wiki-frontmatter-sync`** を起票（CLAUDE.md の仕様駆動ルールに従い code 先行しない）。
- **apply（17/17 タスク、strict valid）**: 新ツール `jooto_frontmatter_sync.py`（frontmatter の `jooto:` ブロック検出、`board_name:`/`last_synced:` 行のみ **in-place 置換**で YAML round-trip なし・本文バイト等価保全、`data/jooto/{board_id}_*` glob + board.json `id` 厳密一致）+ CLI + 15 テスト + `daily_pipeline.sh` Step 1b 配線（jooto-backup 直後・非致命・既定 OFF ゲート）。
- 実データ初回同期（`MeguruWiki@6b13b8f`）: 36 案件中 **32 更新・4 スキップ**（board_id 無し）・warning 0、32 file 各 1 行のみ変更で scope 限定を実証。
- **yq 未導入問題**: apply 中のテストで発覚 — この Mac（LaunchAgent 実行機）に `yq` が無い。`daily_pipeline.sh` の `_cfg()` は yq 依存のため常に既定値を返し、`jooto_frontmatter_sync`・`wiki_refresh`・`wiki_auto_absorb` の config ゲートを ON にできない（healthcheck/weekly_report_frequency は別パーサで無関係）。→ Step 1b 自動化には `brew install yq` が前提。当面は CLI 手動実行。
