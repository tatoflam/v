MeguruPMReport の長丁場・手動フル運用セッション（247 turns）。ソース最新化 → 週次レポート生成 → Milestone 書き戻し + 色塗り → きつね塚誤検知の是正 → republish までを一気通貫で回し、途中で再利用可能な運用ツールと運用取り決めを 2 つ確立した。

## 決定・取り決め（durable）

### 1. 合図フレーズ「フル手動実行して」
手動フルパイプラインを一発で回すための合図を **「フル手動実行して」**（＋任意で `run_date=YYYY-MM-DD`）に固定する取り決めをした。この号令で以下を **この順序・本番フル反映** で回す：

**wiki更新（ingest→triage→absorb）→ 全リソース差分収集（jooto/pm-master）→ 週次レポート生成 → Milestone writeback → Milestone 再取得＋工程グリッド再生成 → publish（最後）→ alert 送信**

- LLM 部分（wiki absorb・週次生成）は Claude が直接やる（headless `claude -p` はネスト対話セッション内ではサンドボックス不可のため）
- **publish は必ず最後（前倒し厳禁）**。既存の「レポート作成して＝1〜9 フルセット」運用と同じ順序制約。
- headless の cost 実測はこの経路に含まない（それは端末/LaunchAgent 側）

### 2. wiki 取得の Gmail 読みと 週次レポート生成の Gmail 読みは「別物」
ユーザーが「Step1 の wiki 取得メール読みと Step3 の週次レポート Gmail ライブ読みは重複では？」と疑問。結論は **重複ではなく 2 つの独立した読み手**。共通点は「両方 Gmail を読む」だけで、目的・クエリ・窓・方式・出力先・寿命がすべて別：

| 軸 | ① wiki 取得（Step 1） | ③ 週次レポート生成（Step 3） |
|---|---|---|
| 目的 | wiki に設計ログを恒久蓄積 | 今週の案件状況を要約してレポート化 |
| クエリ | `to/cc` に **togami-log@ / design@**（配信・ログ用アドレス） | 案件ごとの `gmail_query`（例 `label:案件-せ:千住4丁目`）＝案件ラベル別スレッド |
| 期間窓 | 直近2日（`wiki_refresh_lookback_days`） | 直近8日（`newer_than:8d`） |
| 方式 | gmail-grabber が **.eml をディスク保存**（ingest が実体ファイルを要求） | MCP Gmail で **ライブ読み**（保存しない） |
| 出力先 | ingest→triage→absorb → wiki shard | LLM 要約 → reports/ の案件セクション |
| 寿命 | 長期の wiki（積み上げ） | 今週スナップショットのみ |

母集団はほぼ別集合。同じ1通が両方に写ることはあり得る（design@ 宛かつ案件ラベル付き）が「同じ手紙を別目的の 2 人が読む」だけで無駄な二度手間ではない。1回に統合できない理由＝クエリ別・窓別・方式別（wiki は .eml 実体が必要／レポートは読むだけ）・出力先と寿命が別。

## 成果物（artifact）

### `scripts/wiki_catchup.sh` — 再利用可能な wiki 一括 catch-up runbook（NEW）
日次が長期間止まって wiki が古びたときの復旧、または初回大量取り込み用。daily_pipeline.sh の **Step 0a–0c（gmail取得→ingest→triage）＋ Step 7（auto_absorb）** を **可変ルックバック** で一発 orchestrate する（再実装なし・daily_pipeline と同じ契約）。

- 取得範囲は既定で「前回この runbook が成功した日以降」＝前回からの差分（カーソル `state/wiki_catchup_last_run` 管理、absorb 成功時のみ当日へ更新）。初回はカーソルが無いので `--since` を明示。
- `--model`（既定 config `wiki_absorb_model`。例 sonnet / opus / claude-opus-4-8）、`--since YYYY-MM-DD`、`--chunk-size`、`--mail-user`、`--no-llm`、`--skip-fetch`、`--no-absorb`、`--absorb-only`、`--dry-run`。
- **端末で実行必須**（triage/absorb が `claude -p` を使う。対話 Claude セッション内のネスト claude -p はサンドボックスでブロック）。`WIKI_INGEST_WORKER=1` を export して個人 vault auto-ingest フックを抑止（§0.1 と同じガード）。
- checkpoint 運用（想定外に大量＝firehose のとき）: (1) `--since ... --no-absorb` で DL→ingest→triage で停止 → (2) triage_log の生き残り数確認 → (3) `--since ... --absorb-only --model <sonnet|opus>`。
- 今回の実行意図: wiki の最終 sync が 6/16 で止まっていた（日次 incremental は 2 日窓なので 6/16→今のギャップは埋まらない）ため、`scripts/wiki_catchup.sh --model opus --since 2026-06-16` でバックフィルしてからパイプラインを回す、という判断。
- 注意: この wiki 用 gmail-grabber フローは `wiki_refresh` が長らく false で一度も走っておらず、thomma@ の OAuth 同意が未了だと最初の DL でブロックし得る。
