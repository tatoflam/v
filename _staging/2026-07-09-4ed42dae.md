# meguru-construction.com 問い合わせフォームが無言で消える問題の切り分け

**リポジトリ**: `meguruit/website-theme-meguru`（本番 WordPress サイト meguru-construction.com のカスタムテーマ「meguru」）。対象は `meguru/assets/js/contact_send.js` と `meguru/assets/php/contact_send.php`。cwd = website-theme-meguru。セッション本体は 2026-07-09 夜（JST 19:29〜、翌朝 08:24 にクローズ）。

## 症状
問い合わせが送信されておらず、画面上はアラートも開発者ツール上のエラーも出ず、サンクスページ（`/thanks/`）だけが表示されてトレースできない状態。WordPress 管理者権限あり。

## 根本原因（構造）
`contact_send.php` は **GAS（Google Apps Script）へ HTTP 中継するだけ**のファイル（メール送信も DB 接続も無い）。失敗パスがすべて `echo '通信エラー'; return;` で終わっており、これは **HTTP 200 を返す**。`contact_send.js:84` の jQuery は 200 なら `success` を発火して `/thanks/` へリダイレクトする。したがって GAS 中継が失敗してもユーザーには成功に見え、`.fail()` は呼ばれず、コンソールにも何も出ない。**「失敗しても成功に見える」設計そのものが、発見を長引かせた唯一の原因。**

## 切り分けの手順と実測結果（決定的）
1. **Network タブのレスポンス本文** = `通信エラー`、**TTFB 3.2 秒**。→ `file_get_contents` で外部へ出て帰ってきた時間。つまり `contact_send.php:37`（GAS レスポンスが `status:ok` でない）に到達しており、PHP は正常・GAS 中継が失敗、と確定。ユーザー提示の「mb_send_mail や DB 接続が原因」という当初診断は外れ（このファイルにそれらは存在しない）。
2. **GAS へ GET**（`curl` はローカルで user 実行）→ `200` で GAS 自身のエラーページ `スクリプト関数が見つかりません: doGet`。`accounts.google.com` へのリダイレクトは起きず。→ **デプロイは生存・アクセス権は「全員」のまま**。最有力だった「アクセス権喪失／デプロイ消滅」は消えた。
3. **GAS へ POST** → `{"status":"error","errors":[{"message":"不明なエラーが発生しました"}]}`、`url_effective` は `script.googleusercontent.com/macros/echo?...` まで到達。→ 302 リダイレクトは正しく追われ `doPost` は実行され JSON が返った（`file_get_contents` のリダイレクト再送トラップは今回は起きていない）。**PHP は無罪、GAS の `doPost` 内部でエラー**と確定。`不明なエラーが発生しました` は Google 標準文言ではなく **スクリプト作者が `catch` に書いた独自文言**で、真の例外はそこで握り潰されている。

## 解決（user 側）
> 「GASの権限設定の問題だったので、権限調整とエラーハンドリングをすれば解決する」

GAS 側の権限調整＋エラーハンドリングで解決見込み。GAS 本体のソースはこのリポジトリには無い（リポジトリ全体を grep しても GAS 参照は `contact_send.php:18` のデプロイ URL 一箇所のみ。埋まっているのは**デプロイ ID** `AKfycb...` でありスクリプト ID ではないため、これ単体から逆にプロジェクトを辿る公式手段は無い。探すなら `script.google.com/home`、またはコンテナバインド型ならデータ蓄積先スプレッドシートの「拡張機能→Apps Script」、または過去の通知メール送信元アカウントから辿る）。

## 提案済みだが**未適用**のソース修正（distill 判断用）
チャットで提示のみ。リポジトリへの Write/Edit は行っていない（＝コード変更は本セッションでは未反映）。PR も無し（06_output 証跡なし）。

`contact_send.php`:
- **【最重要】失敗時に HTTP 200 を返す設計を廃止** — 4xx/5xx を返して `.fail()` を発火させる。GAS を直しても、この構造が残れば次に壊れたとき再び無言で問い合わせが消える。
- `count($_POST) < 5` は**オフバイワン**（フィールドは 6 個）。1 つ欠けても検証を通過し `$data['name']` 等が未定義キー参照になる。個別存在チェックに置換。
- `contact_send.php:31` の `'Content-Type: application/json charset=UTF-8'` は**セミコロン欠落の不正ヘッダ**（GAS が寛容で通っていただけ）。`application/json; charset=UTF-8` に修正。
- `file_get_contents` に**タイムアウト未指定**（GAS 無応答時 `default_socket_timeout` 既定 60 秒まで待つ）→ `timeout` を明示（提案は 20 秒）。
- `json_decode` 結果を検証せず `$contents_json['status']` を読んでおり、GAS が HTML を返すと PHP 8 で警告。→ `is_array` チェックを追加。
- `mb_language()` / `mb_internal_encoding()` はメールを送らなくなった今は無意味（削除可）。
- 提案リファクタでは失敗時も `error_log('[contact_send] ...')` でサーバー PHP エラーログに痕跡を残す（このファイルは `wp-load.php` を読まない素の PHP なので `WP_DEBUG_LOG`／`wp-content/debug.log` は効かない点に注意）。

`contact_send.js`:
- `.done()`/`.fail()` に置換し、`.fail()` で `console.error` ＋「送信に失敗しました。お手数ですがお電話にて…」の alert を出し、成功時のみ `/thanks/` へ遷移。

## 再利用可能な学び（05_learn 候補 — distill で分割検討）
- **PHP→GAS 中継で「HTTP 200 + 独自エラー文字列」を返すと、フロントの ajax は success 扱いになり無言で失敗が握り潰される**。中継 PHP は必ず上流の失敗をステータスコードに反映すべき。
- GAS Web アプリの典型的壊れ方: 再デプロイでアクセス権が「全員」から外れる／デプロイ ID 変更で旧 `/exec` が死ぬ／`doPost` の `catch` が例外を独自文言で握り潰す。切り分けは「GET で生存確認 → POST で `status` 確認 → GAS の『実行数』ログ」。
- `file_get_contents` での GAS POST は `script.googleusercontent.com` への 302 を伴い、HTTP ラッパーがリダイレクト時に POST とボディを再送してしまう古典的トラップがあり得る（今回は該当せず）。`allow_url_fopen` が Off だと黙って `false` を返す。堅牢化するなら cURL（`CURLOPT_FOLLOWLOCATION` + `CURLOPT_POSTREDIR`）。
