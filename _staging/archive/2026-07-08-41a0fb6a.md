# ToDoBot 6/12 通知未達インシデント（スケジューラドリフト）→ 恒久修正・汎用リカバリ機能・PR #1 マージ

セッション期間: 2026-06-12〜06-24（cwd: ToDoBot）

## 原因
Cloud Scheduler の `every 15 minutes` は「前回**完了**から N 分後」の**相対スケジュール**で、発火位相が毎日ドリフトする。6/12 に tick が `:07:03`（スロット `:00` から 423 秒）まで後退し、`_due_slot()` の照合窓 `±7分（420秒）` を 3 秒超過 → 06:00 / 18:00 とも**1件も起動しない**サイレント障害（送信失敗ログすら出ない）。最後の正常送信は 6/11 18:00。

## 実施（すべてデプロイ・マージ済み）
1. **再発防止**: スケジュールを固定壁時計の crontab `*/15 * * * *` に変更（ドリフト消滅、照合窓は ±7分のまま据え置き — 固定化で拡張不要と判断）。
2. **汎用リカバリ機能**: `daily_report_trigger` に `?since=&until=&note=&label=` を追加 — 任意の時間窓のメッセージを **1通に集約**し注記バナー付きで再送。`?date=&slot=` での過去日再送も追加。24h はパラメータにすぎず汎用（ハードコードなし）。冪等キー `label` で二重送信防止。
3. **リカバリ実施**: 6/11 18:00 → 6/12 18:00 の 24h 分を集約送信 — 1課 46件 / 2課 13件 / team-b 0件スキップ（正常）。当初 36h と誤認しかけたが、6/13 06:00 便は修正済みスケジューラが自動配信するため 24h が正解（ユーザーの指摘が正しかった）。
4. **ドキュメント**: 新規 `docs/operations/runbook-missed-report.md`（調査＋リカバリ手順＋インシデント記録）、`alerting.md` に**未送信ハートビート・アラート**（「起動しなかったこと」の検出 — 今回の盲点）、README/deploy.md 更新。
5. OpenSpec change `missed-report-recovery` 作成→アーカイブ（`2026-06-24-missed-report-recovery/`、spec sync なしの delta 保管運用を踏襲）。

## 決定事項
- 2026-06-12: Cloud Scheduler の相対スケジュール `every N minutes` は使わず、固定 crontab 表記にする — ドリフトによる照合窓外れを構造的に排除。
- 2026-06-12: 障害時の再送は「スロット単位の再実行」でなく「任意ウィンドウ1通集約＋注記」を標準とする。

## 運用知識（05_learn 候補）
- Cloud Scheduler `every N minutes` は相対スケジュールでドリフトする。時刻照合型の定期ジョブは必ず crontab 固定表記にする。
- 「実行されたが失敗」だけでなく「**そもそも起動しなかった**」を検出するハートビート監視が必要。
