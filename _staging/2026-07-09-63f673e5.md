## 核心の学び：公開サイトの締切の「真実」の在り処
締切値は次の経路で公開サイトに出る（既存メモ [[deadline-lives-in-github-secret]] の具体化）:

**GitHub Actions Secret `DEADLINE`** → デプロイ workflow（`.github/workflows/deploy.yml`, `workflow_dispatch` 有効）→ `build-config.js` が `dist/config.js` を生成 → **Cloudflare Pages** で配信。

- 手元／リポジトリの `config.js` は `.gitignore` 対象で**ローカルプレビュー専用**。編集しても公開サイトには一切影響しない（これが「表示が更新されない」原因だった）。
- Secret は 2026-06-27 設定、最後のデプロイも 6/27。以降 main への push が無く再ビルドされていなかった。

## 締切変更の正しい手順（次回のRunbook）
```bash
printf '2026-07-05' | gh secret set DEADLINE   # ← Secret書き込みは要手動承認（自動モードでガード）
gh workflow run deploy.yml                       # デプロイ起動（30〜40秒で反映）
```
今回のデプロイ run: 28881658207（success・36秒）。https://github.com/tatoflam/minami-gakudo-tshirt/actions/runs/28881658207
- `gh secret set` は自動モードのガードで2回ブロックされた（Secret ストア書き込みは明示承認が必要）。最終的にユーザーが手動で `gh secret set DEADLINE` を実行し、`gh workflow run deploy.yml` は Claude が実行。
- 恒久対応案：settings の permissions に `gh secret set` を許可すれば次回から Claude が直接実行可（Secret常時書き込み許可の留意点あり）。

## clasp run が使えない件（再確認・既存メモの裏取り [[clasp-run-not-usable]]）
backend の Script Property を clasp で更新しようとして失敗。試した経路と結果:
| 経路 | 結果 |
|---|---|
| `clasp run`（manifest に executionApi 追加＋Apps Script API 有効化） | 権限エラー。既定ログイントークンが manifest スコープ（gmail.send/spreadsheets 等）を網羅せず `scripts.run` が認可拒否。解消には独自 OAuth クライアント＋GCP プロジェクト紐付けが必要で過剰 |
| manifest の oauthScopes 最小化で回避 | セーフガードがブロック（正当）。公開 `@HEAD` web app を壊すリスク |
| web app 経由で実行 | 稼働中の `@3` 本番デプロイ（実フォーム）に触れるため、締切当日は見送り |
- 手動フォールバック：スクリプトエディタで `PropertiesService.getScriptProperties().setProperty('DEADLINE', '2026-07-05')` を1回実行 → 削除。ただし今回ユーザーはエディタにアクセスできず、汎用文言のままで確定。
- Apps Script プロジェクトは一時関数・manifest の一時変更をすべて撤去しクリーンな状態に復元、git ツリーもクリーン。
