## 1. add-github-ci-site-deploy（propose → apply → 本番稼働 → archive）

**目的**: 手動 firebase deploy だった週次レポート配信を「main へ commit&push → GitHub Actions が Firebase にデプロイ」に切替。さらに MeguruWiki の最新コンテンツも同一サイト（同じ GWS ドメインゲート）に `/w/**` として配信。

**設計決定（ユーザー回答で確定）**:
- ブランチ保護なし（開発/運用とも単線）→ bypass 設計不要
- 認証は **SA（Service Account）第一経路**、`FIREBASE_TOKEN` は障害時フォールバックに格下げ
- wiki allow-list は `shards/**` + `docs/**`（`config/wiki_site.yaml`、default-deny、ルート直下は初期スコープ外）
- wiki push は daily のため `repository_dispatch` が主経路、schedule はフォールバック

**実装物**:
- `site/functions/reports/`（サニタイズ済み payload 16 週分）を git 追跡化 — main が CI デプロイの単一情報源
- `.github/workflows/deploy-site.yml`（main push / wiki dispatch / schedule トリガー）
- publish CLI に `--deploy-mode ci|local`、`scripts/publish_commit_push.sh`（一時 worktree 経由で payload のみ commit&push）
- MeguruWiki 側 `notify-site-deploy` workflow → `repository_dispatch` → deploy-site の連鎖
- wiki render は CI 内で実行（`site/functions/wiki/` は gitignore）。frontmatter 用に PyYAML を正式依存化

**GCP セットアップ（runbook = `openspec/changes/archive/2026-07-16-add-github-ci-site-deploy/setup-runbook.md`）**:
- SA: `firebase-ci-deployer@meguru-pm-report-ad6b1.iam.gserviceaccount.com`
- ロール **4 点**必要: `roles/firebasehosting.admin` / `roles/cloudfunctions.developer` / `roles/iam.serviceAccountUser` + **`roles/firebase.viewer`**（デプロイ時の firebase CLI がプロジェクト参照に要求、後から判明）
- **Cloud Billing API の有効化も必要**（デプロイ途中で要求され、伝播待ちも発生）
- 鍵 JSON: `~/.config/meguru-plugins/firebase-ci-deployer.json`（600）
- GitHub Secrets: `FIREBASE_SERVICE_ACCOUNT` / `FIREBASE_WEB_CONFIG`、MeguruWiki 側に `SITE_DISPATCH_TOKEN`（PAT）

**CI デバッグで踏んだ障害（計 6 回試行で成功）**:
1. workflow の pip install に `linkify-it-py` 欠落 → wiki render 失敗
2. **`FIREBASE_SERVICE_ACCOUNT` secret が空登録** — 鍵 JSON ファイルが作成後に 0 バイト化（truncate）していたのが原因。鍵再発行 + secret 再登録 + 旧鍵は SA から削除
3. `npm ci` の lockfile 問題
4. `roles/firebase.viewer` 不足
5. Cloud Billing API 未有効

**検証結果**: deploy-site 成功（2m56s）、wiki 連鎖 end-to-end 成功（1m3s）、`publish --deploy-mode ci` → `publish_commit_push.sh` が**差分ゼロ no-op 終了**（一括サニタイズ済み payload とレンダラ出力の一致 = 収束性の証明）。マスキング監査クリーン（144 ページ、未マスクのメール・金額ゼロ）。

## 3. デグレ事故と publish ガードの進化（重要な教訓）

**事故**: 検証で `publish --run-date 2026-07-15` を再実行したため、7/15 の会議中に Google Doc へ入れた人編集が機械版で上書きされた。publish は同一 run_date の Doc をローカル Markdown から冪等上書きする設計だったため。復旧は Google Docs のバージョン履歴から復元（baseline-sync 前の復元が必須 — 次回生成が復元後内容を前回比較元に取り込むため）。

**第 1 弾 `add-publish-doc-edit-guard`**: Doc 更新前に 1 回 export し配信時スナップショットと空白正規化比較 → 人編集検出で**中止**。`--force-overwrite-doc` で退避付き上書き。実 Doc で偽陽性なしを実測して archive。

**第 2 弾 `merge-doc-edits-on-republish`**（ユーザー要望: 中止ではなくマージ）:
- 人編集検出時、案件セクション内の挿入行を新コンテンツ同一案件末尾へ `- 手動追記:` ブロックとしてマージ — **generation-time carry-forward と同じ抽出器 `compute_carryforward` を再利用**（人編集の扱いが翌日生成と同一セマンティクス）
- マージ行に sanitize 適用（生の金額・メールも配信物では伏字）、正規化一致で重複排除（再々配信でも冪等）
- 機械行の書換え・削除は機械が勝ち（記録のみ）、案件セクション外の編集はマージ対象外。監査は `state/publish_merge_{run_date}.json`（gitignored）
- テスト: plugin 142 / pipeline dry 60 通過。archive 済み

- 結果: **生成 → publish → main push → CI デプロイの headless フルサイクル初完走**（`chore(publish): weekly report 2026-07-16`、CI 2m54s）。復元済み 7/15 Doc の会議編集は carry-forward で 7/16 レポートに反映

## 6. /pm-status と /wiki-status の混同（Claude Code の学び）

- ユーザーが `/pm-status` を打ったつもりが `/claude-wiki:wiki-status`（個人 Obsidian vault `tatoflam/v` 用の個人プラグイン）に曖昧一致で解決され、無関係な個人 wiki の状態が表示されて混乱
- 原因: `.claude/commands/pm-status.md` はセッション中に新設されたため、**先に起動していた CLI セッションの補完リストに未登録** → 新セッションで解消。`.claude/commands/*.md` は置くだけでプロジェクト固有スラッシュコマンドになる（plugin 化は複数リポ配布時のみ）
- ついでに `/pm-status` の absorb commit 誤検出を修正（`[auto]`/`[weekly]` を commit subject 先頭一致に限定、commit `db3f9d3`）
- 「wiki」が 2 系統ある整理: 個人 wiki（tatoflam/v、/wiki-ingest 等）と業務 wiki（meguruit/MeguruWiki、daily Step 0/7 + `/w/**` 配信）
