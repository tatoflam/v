# consolidate-project-identity: 設計 → 実装 → 5リポ横断 PR → archive まで完結

MeguruPMReport の週次パイプラインで長年並存していた **2 つの案件同定キー**
（`config/projects.csv` の FY26 ラベル ＋ MeguruWiki shard の resolver-slug）を、
identity 単一マスタへ寄せる `consolidate-project-identity` を起票・実装・archive
まで一気通貫でクローズしたセッション（2026-07-11〜07-14、UTC）。

## 発端

- 冒頭の問い「未実装の propose はあとどういう内容だっけ？」に対し、未 archive の
  OpenSpec change が **2 件**（`add-triage-routing-enrichment` 36 タスク未着手 /
  `add-wiki-absorb-orchestration`）と確認。いずれも [[project_consolidation_resume]]
  にある consolidate-grabber-plugins（archive 済）の後継。
- そこから「MeguruWiki の shard を Claude が判断する方式」と「projects.csv 参照方式」が
  **統合時に競合する**（2 キー問題）という設計論に発展。ユーザーが **B案**
  （identity/routing 単一マスタ化を独立 change として起票）を選択。

## 用語の確定（江古田江原の位置づけ）

- 江古田江原（= `FY26_21`）を spec で繰り返し例示していたのは、**呼称と slug が字面で
  全く重ならない最悪ケース**だから。メール/Jooto 呼称 `江古田江原` ↔ resolver slug
  `NakanoEhara3Chome`（中野区江原町3丁目）。
- ニュアンス確定: 「slug は shard フォルダ名としては安定だが、人間の呼称からは推測不能
  なので **alias 表がないと join が切れる**」。千住4丁目→Senju4Chome 等は字面が透ける
  ので、あえて最悪ケースを scenario に採った。

## 実データ突合による重要な設計判断（← このセッションの核心的発見）

「slug を唯一の主キーに単一キー化する運用移行は検証済みか？」→ **未検証だった。実行して
検証した結果、"今のまま単一キー化するのは危険" と明確に出た。**

`config/projects.csv` 27 行 × `_alias_resolver.md` の突合結果:

| 区分 | 件数 | 中身 |
|---|---|---|
| exact 一致 | 19 | 問題なく slug 解決（江古田江原含む） |
| partial（要人手確認） | 1 | 東尾久4丁目 → `HigashiOgu`（resolver 側は「東尾久」、4丁目が同一案件か別サブか要確認の偽陽性候補） |
| **GAP（shard 不在）** | **7** | 堀ノ内2丁目 / 大久保2丁目 / 西永福 / 東十条3丁目 / 東大附属中前 / 南馬込1丁目 / 中野区本町6丁目 |

- **能動 27 案件のうち約 30%（8件）が現時点で slug に単一キー化できない。**
- 根本原因: GAP 7 件は新しめの FY26 案件。**Gmail ラベルは人が既に 27/27 付与済**だが、
  wiki shard がまだ absorb されていない（Claude が未読）。つまり **今日時点では明示キー
  （Gmail ラベル）の方が slug より網羅的**（ラベル 27/27 ＞ slug 19〜20/27）。
- これは add-wiki-absorb-orchestration が解こうとしている「absorb が人力頼みで滞留」
  問題そのもの。**「slug が完成する ← absorb が走る ← 新案件が来る」の鶏卵関係**があり、
  slug を先に唯一キーにすると新案件が毎回こぼれる。

### 設計への反映（採用した緩和）

1. **D1「slug を唯一の主キー」→「slug を join key（nullable FK）」に緩める。**
2. collection/routing の実キーは **Gmail ラベルのまま**（人所有・完全・安定、捨てない）。
3. overlay（projects.csv）は自分の行 PK（FY26/ラベル）を保持、slug 列は**空を許す FK**。
   → GAP 8 件も overlay に存在でき、shard ができ次第 slug が埋まる。
4. 重複排除目標は達成: display_name / gmail 同定情報は resolver へ寄せる。FY26 は join 表
   のローカル PK に格下げ、identity を主張しない → 2 キー問題は解消。
5. fallback（`## routing/` 未定義 → ラベル直指定）は "optional" でなく **"load-bearing"**
   （移行時に 30% がこれに依存）。
6. **slug 必須化は gap=0 になってから。** 移行ツールは 8 GAP を「stub shard 作成 PR」として
   提案（Claude 提案／人マージ）、埋まるまで単一キー強制はしない。

## 実装・cutover

- `/opsx:apply consolidate-project-identity` で実装。`config/projects.csv.slug-preview`
  を生成 → **ピボット承認 → preview 目視 → `--apply` → 5.6 docs** の順で cutover。
- 東尾久4丁目 = `HigashiOgu` を人手確認で確定（partial 昇格）、MeguruWiki 側に alias 反映。
- 後続 2 件（triage / wiki-absorb）の依存記述を軽く先行修正。
- resolve_slug 候補ランキング化（旧 5.0/5.6/5.7）は `add-triage-routing-enrichment` へ
  畳み込み（別 change に集約）。
