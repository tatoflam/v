### apply [2] add-bff-guard — 27/29 で archive (commit `637fabe`)

5 値 Verdict ステートマシン (PASS / BLOCK / REWRITE / DOWNGRADE / HOLD) を Review BOX 直後に挿入。Guard は **必ず 1 ターンに 1 回** 走る (PASS 含めて Verdict 記録)。

**実装の核**:

| 機能 | 実装 |
|---|---|
| **5 値 Verdict** | `GuardEvaluator.evaluate()` が PASS / BLOCK / REWRITE / DOWNGRADE / HOLD を返す純関数。draft + attunement output を受けてルールベース判定 |
| **max_rewrite 制御** | `MAX_REWRITE` env 変数 (既定 2) を超えた次の非 PASS Verdict は強制 HOLD コアサ。`rewriteCount >= MAX_REWRITE` で Drafting LLM 呼び出しを必ず止める |
| **HOLD は LLM ゼロ** | `pickHoldTemplate()` 純関数で Presence/Mirror 級のテンプレ文言を返す。LLM 呼び出し不要なので攻撃耐性 + コスト低 |
| **DOWNGRADE は attunement 1 段下げ** | `lowerByOne()` で attunement の `chosen_level` を 1 段下げ → `[INTERVENTION_LEVEL]` タグを再注入して Drafting に戻す。これで Drafting プロンプトを差分修正 |
| **BLOCK は 422 + reason** | `GuardBlockError` (`AppError` 継承) を throw。`error-handler` が `reason` を response body に出す |
| **構造化 ViolationLog** | pino warn で `{ type, count_in_session, attunement_level, action, rewrite_count, draft_excerpt? }`。本番では `excerpt` 省略 (PII 漏洩防止) |
| **PM Chat UI** | Guard セクションに verdict バッジ + 4 項目表示。mock Pattern C で HOLD 視認可 |

### 学び (横展開可能)

- **explore → apply の橋渡しは「順序確定 + 各 change のスコープ判定」まで**: explore で「A → B → C → D の直列が最適」と決めて apply に渡したが、`/opsx:explore` 内で **stance** が「実装しない」になっているので、依存解析と「どこまでが本 change のスコープ / どこからが別 change」の境界判断まで explore 側で済ませると、apply 側はタスクを潰すだけになって滑らかに進む
- **`§5.2 PM Chat 目視` のような「ローカル E2E が成立しない」タスクは A/B/C 三択で user に決めさせる**: B (SAM Local 完全対応) は env.json 整備 + habi-identity ローカル起動という別 change 級の地ならし、C (dev デプロイ後確認) は §7.1 + auth 復旧前提。A (mock UI 確認) は handler test §4.3 で実ロジック検証済みなので「UI 描画のみ確認」で精神を満たせる。**「短期 workaround + back-fill 条件明記」が一番タスクが進む**
- **deferred タスクは tasks.md に back-fill 条件を明記して archive する**: archive 後も deferred 項目が見えるように、tasks.md にチェックボックス + 条件 (e.g. 「habi-os PR 完了後」「dev deploy 後」「quality-always-on 完了後」) を明記。これで spec の「実装完了」と change の「全タスク完了」が乖離しても追跡可能
- **5 値 Verdict は 3 値 (PASS/REWRITE/HOLD) より細かいが意味がある**: BLOCK = 即返却 (再 Drafting しない)、REWRITE = 同一 attunement で再 Drafting、DOWNGRADE = attunement 1 段下げて再 Drafting、HOLD = テンプレ返却 (LLM 呼ばず) という 4 つの非 PASS が **コスト・遅延・安全性で異なるプロファイル** を持つので、3 値に圧縮すると DOWNGRADE / HOLD / BLOCK のどれかが他に丸められて意図がブレる
- **HOLD テンプレを純関数で返す設計**: HOLD で LLM を呼ばないことで (1) 失敗モードでさらに失敗を重ねるリスクを断ち切る、(2) コスト発生ゼロ、(3) Presence/Mirror 級テンプレなので人格 OS の "そばに居る" 振る舞いを崩さない。ガードレール設計として参考になる pattern

- **subject 軸で広げる > 肩書軸で広げる**: 「複数カテゴリを横断したい」とき、肩書 (デザイナー、専門家) で広げると教え口調にズレる。subject (私の家、私の体験) で広げると一人称が全カテゴリで成立し、差別化も "私の" 私性で取れる
- **electronic isolation は device 軸が boundaries の最弱点**: 1 デバイスで 3 アカ運用すると IP/SIM/カウント番号で isolation できても、device fingerprint (フォント / VPN history / cookie spillover) が同居する。mitigation は別ブラウザ profile (Chrome / Firefox / Brave) + Threads web 化。コストとリスクのバランスで「完全 isolation を狙わず実用的 best effort」に倒した
- **段階的 scaling = 1 アカ内 protocol を複数アカ間に拡張**: warmup の「観察 → 段階的に活発化」を 1 アカ内ではなく複数アカ間にも適用する発想。「前アカが BAN ゼロ + 収益軌道」を milestone gate に置くと、1 アカ目で踏んだ罠が 2 アカ目で再発しないことが事前に分かる
