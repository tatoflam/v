### 学び (横展開可能)

- **混在 working tree の整理は user に 3 択提示 → 順序実行**: 「1 commit にまとめる / change 単位で分離 / 別 PR で切る」を提示 → 「1, 2, 3 の順で」で 3 commit に分離。auto モードで黙って 1 commit にしていたら playbook-process-revamp の文脈が失われていた。**git history 上の change boundary は spec の change boundary と一致させる**価値が大きい
- **PII scrub は memory + design の slug 確定が前提**: `gut-health-immune-allergy` を default として memory・design 両方に置いておいたので grep 対象が一意に決まり、handle / amazon_tag のみ redact で済んだ。design / memory がバラバラだと scrub 漏れリスクが跳ね上がる
- **GH secrets 削除は dispatch のテストではなく cron run で確認**: 削除直後 `gh workflow run weekly_report.yml` を蹴っても sync step は元から無いので意味なし。次月曜の本番 cron で「sync 不要 + 成功」を確認するほうが本質。**「設定削除の検証は実運用 cron で取る」** という運用 pattern
- **大型 archive は 41 renames でも git tree が壊れない**: `git mv` を逐次実行 → status で 41 renames のまま捕捉できた。Git の rename detection は同名ファイルが別ディレクトリへ移動するパターンに強い。**archive 操作は `mv` ではなく `git mv` を使う**
- **`auto` モードでも user 入力待ちタスクは明示的に停止する**: token 配置 / dry-run 実機検証 / async monitor は Claude が代行できない。auto モードの定義を「Claude が安全に実行できる範囲」と user が共有していると、停止地点で confusion が起きない
