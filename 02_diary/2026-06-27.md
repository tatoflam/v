### e702af57 — land-searcher REINS WAF が HeadlessChrome を 503 で弾く問題の根治 (meguruit/land-searcher、2026-06-26→27、~15h 跨ぎ)

GitHub Actions の REINS 巡回が 503 で落ち続けていた CI を、UA matrix probe で原因
切り分け → Selenium の UA を通常 Chrome に上書きして根治。`origin/main` に
5 commit push 完了。

**診断手順 (UA matrix probe):**

| UA | 結果 |
|---|---|
| curl / urllib | 200 |
| 通常 Chrome (デスクトップ) | 200 |
| `HeadlessChrome/<ver>` (Selenium 既定) | **503** |

→ WAF が UA パターンマッチで `HeadlessChrome` を block している、と確定。

**修正:**
- `7bdc449` Fix REINS WAF bypass by overriding headless Chrome User-Agent —
  WebDriver 起動オプションで通常 Chrome UA を指定
- `235b479` Restore CI run with new headless mode, align Service Account key filename
- `c79bf69` Revert temp diagnostic (page-state log + screenshot) — UA matrix probe
  仮設コードの片付け

**学び (汎用化):**
- WAF は IP やレートだけでなく **UA パターン**で bot を判定する。Headless ブラウザを
  業務利用するなら UA は最初から Chrome 風に上書きするのが defensive な既定値
- `curl / urllib / Chrome / HeadlessChrome` の 4 種を並べる UA matrix probe は WAF
  起因切り分けの最短手順 (= 503/200 の境界を一発で出す)

→ 新規 [[03_work/land-searcher]] / [[05_learn/reins-headlesschrome-ua-block]] を

## 設計上の学び

- **状態の置き場 = 業務粒度に合わせる**: 発注・入荷・受渡・連絡は実際には種類×サイズ
  で起きる。申込単位で持つと「部分入荷」を表現できず、便ごとの再発送・重複連絡防止に
  常に補助フラグが必要になる。最初から明細単位に揃えると `申込` シートは派生サマリで済む
- **「フォームを開いたまま発注確定する」を成立させる装置 = 発注便**: 締切と発注を
  分離。締切は周知 UI 上の目印、発注は便番号で論理的に区切る運用に
- **CC ADMIN_EMAILS の集約**: 送信系コードに散在しがちな宛先を `Config.gs` 1 か所に
  集約 → 名簿変更時の漏れリスクを構造的に消す
