# FY サイクルを使った MM/DD 日付列の年推定

## Summary

業務スプレッドシートにありがちな「**年なしの `MM/DD`**」列を、**既知のフェーズ順序**（＝FY サイクル）と**基準日（today）** から YYYY-MM-DD に復元する方法。純粋な文字列比較（`"9/10" > "3/2"`）で誤判定が出た場合や、cross-year 案件（12 月→1 月の年跨ぎ）を正しく扱う必要があるときに使う。

## Details

### 起きる問題

- 統合表の 8 マイルストーン（`設計開始` / `基本図承認` / `仮受予定` / `本受予定` / `下付予定` / `契約上の着工` / `実着工予定日` / `引渡日`）は `MM/DD` だけで入力されており、年は別途スプレッドシートのフェーズや案件名（FY26_\*\*）から人間が脳内補完していた
- そのまま文字列ソートすると `9/10 > 3/2` と扱われ、過去/未来/overdue 判定がすべて壊れる
- さらに **設計開始が空** の案件もあり、単純に「今年の日付として扱う」では足りない

### 推定ロジック（Meguru 向け実装）

1. **`設計開始` を過去側に固定**: `設計開始` の月日を、today 以前になる最新の年で解釈（`today = 2026-04-25` なら `9/10` → 2025-09-10、`3/2` → 2026-03-02）
2. **以降のマイルストーンは、前フェーズから月が減ったら年+1**: たとえば `下付予定 12/20` の次 `契約上の着工 1/15` は +1 年
3. **設計開始が空のとき**: 最初の非空マイルストーンを「未来側」で anchor（＝today 以降になる最も近い年で解釈）し、それより前のフェーズは適用対象外
4. **データ品質メモ**: 統合表の多くの「次マイルストーン日付」が過去日のまま放置されているケース、評点 = 0（未入力）のケースは `data_quality_notes` として report に残す

### 検証結果（2026-04-25 時点で合っていた例）

- FY26_02 笹塚方南: `下付予定 4/27` → 2026-04-27（2 日後、実際の予定と一致）
- FY26_18 東長崎4丁目: `仮受予定 4/27` → 2026-04-27（実際の 4/27 仮受と一致）
- FY26_01 高円寺南5丁目（鈴木）: 文字列比較では誤判定されていた並行案件カウントが復帰

### 並行案件カウントと「日付未設定」

- 日付が埋まっていない案件でも「意匠設計担当が何らかの案件を抱えている」ものとみなし、**(日付未設定) 注記付きで担当の並行案件に加算**するようにしたところ、加納（FY26_08 + 11）が新規登場、井上が 3 件、鈴木が 4 件で警告ライン到達
- 「未設定 = カウント外」にすると、設計キックオフ前の案件を振り分けられている担当が実態より軽く見えてしまうため

### 横展開の注意

- この推定は **サイクル順序が一方向で厳守される** ことが前提。リスケで順番が前後する業務ドメインには適用できない
- FY 境界（4/1 切替など）は `設計開始` を過去側に固定するルールで自然に吸収されるが、**FY 跨ぎ案件** が多い場合は anchor の初期化方針を案件名（FY26\_\*\*）側から引くのが安全
- 現状はレポート生成側でアドホック変換している。恒久化するなら `plugins/pm-master-grabber/` 側で `integration.json` 行に変換済み日付を同梱するのが筋（follow-up change 候補）
