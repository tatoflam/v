# meguru-appcheck-advisory

## Summary

大塚商会から株式会社めぐるに提案された **AppCheck Pro**（CheckMAL/JSecurity 製、Windows 向けランサムウェア対策ソフト + 自動ロールバック）の導入評価を 2026-06-22 にレポート化。最終結論は **「原則 導入不要」**、HTML 1 枚で納品（[[06_output/2026-06#html-report--appcheck-導入評価レポート-2026-06-22]]）。

## 結論

**重要ファイルを Google Drive の権限管理共有フォルダに置く運用が守られていれば、AppCheck Pro は導入不要。** 徹底困難な少数（4 名想定: 牛山 / 青木 / 大神 / 石田）のみ保険として限定検討。

## 既存スタックでカバーされている範囲

| 機能 | 担っているもの | AppCheck 上乗せ価値 |
|------|----------------|----|
| 侵入防御・実行ブロック | CrowdStrike Falcon Prevent | 重複（増分ほぼゼロ） |
| EDR / 検知 | Falcon Insight | 重複 |
| Drive 上ファイルの復旧性 | Google Drive 版管理（**100 版 _または_ 30 日**、いずれか早い方）＋ ゴミ箱（25 日）＋ 管理者復元 | 重複（追加費用なし） |
| ローカル編集中ファイルの自動ロールバック | （カバーなし） | AppCheck の唯一の固有価値 |

→ AppCheck の固有価値は「**ローカルで編集中、まだ Drive 同期されていない手元ファイル**」の自動ロールバック 1 点のみ。これは運用ルール（Drive 上で編集、長期ローカル保持を避ける）で消せる穴。

## Drive 版管理の運用上の十分性（user の指摘で確定）

100 版は実運用で使い切らない。効くのは **「版数」より「時間」**：

- 非 Google 形式（Word / Excel / PDF）は **100 版 _または_ 30 日** で自動削除
- ランサム復旧の成否は「**被害に気づいて戻すまでが 30 日 + ゴミ箱 25 日に収まるか**」
- 通常運用なら気づける → **無期限保持 / Vault などの追加設定は不要**

「権限限定」だけでは復旧性の根拠にならない（= アクセス制御の話）。説明根拠にすべきは「**Drive の版管理 + ゴミ箱 + 管理者復元**」の方。

## コスト試算（参考・オープン価格）

| シナリオ | 構成 | 年額（概算） |
|---|---|---|
| **A. 限定パイロット**（推奨されない、保険） | クライアント 4 台のみ、CMS なし | 約 ¥38,400 |
| B. 全社展開 | クライアント 50 台 | 約 ¥480,000（非推奨） |

サーバー版（約 ¥36 万 / 台）・CMS Cloud（約 ¥9 万 / 年）は本構成では **対象外**：ファイル管理は Drive クラウド、4 台規模なら手動管理で十分。

## 大塚商会あての確認事項（レポート §8）

導入を進める場合に確定が必要な項目（オープン価格ベースの想定をクローズにする）:

1. クライアント 4 ライセンスの確定見積（数量割引、複数年契約、構築費）
2. CrowdStrike Falcon との共存実績 / 推奨除外設定
3. ロールバック領域の容量上限・保持仕様

## 留意点

- AppCheck は **Windows 専用**（Mac 対象外）
- CrowdStrike の「ロールバック非搭載」記載は公開記事ベースの一般情報、契約モジュール構成で差がある可能性
- 価格はすべて **オープン価格の参考値**

## 提案論理の進化（議論ラリーの記録）

最終結論「原則 不要」に到達するまでの段階的な見直し（user の指摘で 3 回方向修正）:

1. 初稿: 「限定パイロット推奨」（CrowdStrike と検知重複 + Drive で代替可能なので全社一律は非推奨）
2. user 指摘「対象は営業 4 名のみ」→ クライアント 4 台想定にスケール縮小、サーバー不要明記
3. user 指摘「バックアップは Drive 権限管理で担保している」→ 区別誤りの訂正：「権限限定 = アクセス制御」「Drive 版管理 = 復旧性」の 2 軸を分離、後者を説明根拠に据え直す
4. user 指摘「100 版で運用上十分」→ 無期限保持・Vault 推奨を撤回、現状追認の書き方に
5. user 指摘「Drive 共有フォルダ運用なら AppCheck 不要では？」→ 結論を「原則 不要 / 例外として保険」に振り直し

→ ベンダー提案を CrowdStrike + Drive 既存スタックと突き合わせる際、「**検知レイヤー（重複しがち）/ 復旧性（運用で代替可）/ ローカル編集中ファイル（運用ルールで穴を消せる）**」の 3 分解が判断の軸として有効、というパターン。
