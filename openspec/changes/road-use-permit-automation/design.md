# Design — 道路使用許可 窓口申請 効率化ツール

## Context

- 道路使用許可は管轄警察署の窓口で **2週間分ずつ** しか取得できず、必要期間中は「窓口受領 → 同時に次回申請」のループを現場×工種ごとに回す。
- 2回目以降の申請書は「1回目のデータから期間の日付だけ変更」したもの。加えて図面セット（案内図・保安図・迂回路図）と、前回受理済み申請書の写しが必要。
- 申請書様式は警察様式「別記様式第六（第十条関係）道路使用許可申請書」。転記対象は黄色ハイライトの **期間欄（開始年月日時 〜 終了年月日時）** のみ。冒頭の提出日は窓口で手書き、バーコードは固定。
- 実行環境は株式会社めぐるの GWS 組織アカウント。運用者は秘書課／バックオフィス（管理表入力）と現場監督（図面格納・LINE 報告）。
- 制約: 極力軽量にする。外部サーバ・DB・有料 SaaS は使わず、GAS + Spreadsheet + Drive + Docs + Gmail で完結させる。

## Goals / Non-Goals

**Goals:**

- 管理表への期間指定だけで、2週間スロット分割と全スロット分の申請書ドラフト（Google Document）生成が自動で行われること。
- 受領・次回申請のステータスと必要資料（図面セット等）の格納状況が管理表上で一目で分かること。
- 受領日未入力のスロットに対し、始期日の数日前にメールアラートが届くこと。
- GAS の時間主導型トリガーのみで動作し、手動運用は「管理表への日付入力」と「図面の Drive 格納」に限定されること。

**Non-Goals:**

- LINE Messaging API による報告の自動取り込み（将来拡張。当面は人手入力）。
- eGov オンライン申請との連携・再開。
- 手数料の会計処理、押印済み書類のスキャン自動化、印刷の自動実行。
- Word（.docx）形式での出力。Google Document で生成し、印刷はブラウザ／Docs から行う。

## Decisions

### D1. 実行基盤: GAS 単独（スタンドアロンプロジェクト + 時間主導型トリガー）

- **採用**: 組織アカウント所有のスタンドアロン GAS プロジェクト1本。日次トリガー（巡回・アラート）と、管理表の onEdit / メニュー実行（スロット展開・ドキュメント生成）を組み合わせる。
- **代替案**: Cloud Functions / Cloud Run — 過剰。認証・デプロイ・費用管理が増え「軽量」の要件に反する。AppSheet — 帳票生成の自由度が低い。
- **理由**: 必要な API（Spreadsheet/Docs/Drive/Gmail/Trigger）がすべて GAS 標準サービスで完結し、追加費用ゼロ・運用対象は Apps Script エディタのみ。

### D2. データモデル: 1冊のスプレッドシートに「案件マスタ」「申請スロット」「設定」「通知ログ」の4シート

- **案件マスタ (`sites`)**: 1行 = 現場×工種。列: 案件ID / 現場名 / 工種 / 申請期間開始 / 申請期間終了 / 担当課 / 管轄警察署 / 責任者（現場監督）/ 通知先メール / 工種フォルダURL（自動記入）/ ステータス。
- **申請スロット (`slots`)**: 1行 = 2週間スロット。列: スロットID / 案件ID / 回次 / スロット開始日時 / スロット終了日時 / 申請書DocURL（自動記入）/ 図面セット状況（自動記入）/ 前回写し状況（自動記入）/ 受領日（手入力）/ 次回申請日（手入力）/ アラート状態。
- **設定 (`config`)**: アラート先行日数（既定3）、テンプレートDoc ID、ルートフォルダID、管理者メール等のキー・バリュー。
- **通知ログ (`notifications`)**: 送信日時 / スロットID / 種別 / 宛先。重複送信防止の照合に使う。
- **代替案**: 案件ごとに別ブック — 集計・巡回が煩雑になるため不採用。スロットを案件マスタの横持ち列にする — スロット数が可変（工期次第）なので縦持ちが正。
- **理由**: 正規化した縦持ち2表なら GAS の走査が単純で、フィルタ表示だけで「現場別の管理表」としてそのまま使える。

### D3. スロット分割ロジック: 開始日から14日刻み、終端は申請期間終了日で切り詰め

- 申請期間 [S, E] に対し、スロット i は [S + 14i, min(S + 14(i+1) − 1, E)]。時刻は案件マスタの既定（例 0:00〜24:00 相当の運用値）を用い、期間欄には「YYYY年M月D日 H時 から YYYY年M月D日 H時 まで」で転記。
- 再実行時は既存スロットを ID で照合し、**生成済みスロット・Doc は再生成しない**（冪等）。期間延長時は末尾に追加のみ。期間短縮は自動削除せず管理表上で警告表示（誤削除防止）。

### D4. 申請書生成: Google Docs テンプレート + プレースホルダ置換

- `doc03.docx` を一度 Google Document に変換し、期間欄に `{{PERIOD_FROM}}` / `{{PERIOD_TO}}`（必要に応じ `{{SITE_NAME}}` 等）のプレースホルダを埋めたものを**テンプレート Doc** として管理する。バーコード画像・レイアウトは変換後に人手で1回だけ体裁確認する。
- 生成時は `DriveApp.getFileById(templateId).makeCopy()` → `DocumentApp` の `replaceText()` で置換 → 工種フォルダへ移動。ファイル名は `道路使用許可申請書_{現場名}_{工種}_{第n回}_{開始日}-{終了日}`。
- 提出日欄は空欄のまま（手書き運用）、バーコードは複製されるだけで変更しない。
- **代替案**: Docs API の batchUpdate — GAS 標準の DocumentApp で足りるため不採用。Word (.docx) のまま置換 — GAS 単独では XML 操作が重く、印刷も Docs で足りる。
- **リスク**: docx→Docs 変換でレイアウト（罫線・バーコード位置）が崩れる可能性 → 初回セットアップでテンプレートの目視確認・調整を必須手順とする（tasks に含める）。崩れが許容できない場合のフォールバックとして「期間欄のみのラベルシール印字」または「Docs で様式を再作成」を検討する。

### D5. フォルダ構成と資料巡回

- ルートフォルダ配下に `現場名/工種/` を案件マスタ登録時に自動作成し、さらに `01_申請書` `02_図面セット` `03_受理済写し` のサブフォルダを作る。
- 日次トリガーで各スロットについて: `02_図面セット` にファイルが3点以上あるか（案内図・保安図・迂回路図）、`03_受理済写し` に前回スロットの写しがあるか、をファイル数・命名で簡易判定し、`slots` シートに ✅/⚠️ を書き戻す。
- **判定は厳密解析しない**（PDF の中身までは見ない）。あくまで「格納されているか」のステータス可視化に留める。

### D6. アラート: 日次トリガー + 営業日ベースの先行日数 + 通知ログで冪等化

- 毎営業日朝（例 8:00）に実行。各スロットについて「スロット開始日 − アラート先行日数（営業日、既定3）≦ 今日」かつ「受領日が空欄」なら、案件の通知先メール＋管理者へ GmailApp で送信。
- 土日祝は `config` の祝日カレンダー（CalendarApp の日本の祝日カレンダー）で除外。
- 同一スロット・同一種別の通知は `notifications` ログと照合し、**1日1回まで**・解消（受領日入力）で停止。エスカレーション（開始日超過後は件名を強調）を含む。

### D7. 権限・共有

- GAS・管理表・ルートフォルダはすべて組織アカウント（サービス運用用の共有アカウント推奨）が所有。現場監督には工種フォルダを「編集者」、管理表は秘書課・バックオフィスのみ編集可、現場監督は閲覧可。
- GAS は所有アカウント権限で実行（インストーラブルトリガー）。外部公開 Web アプリは作らない。

## Risks / Trade-offs

- [docx→Docs 変換で様式レイアウトが崩れ、警察窓口で受理されない] → 初回に変換品質を確認する受入手順を必須化。崩れる場合は Docs 上で様式を再作成（1回限りの作業）。
- [受領日入力が人手（LINE 報告→転記）のため遅延し、誤アラートが出る] → アラートは「未入力の可能性あり」のトーンにし、管理表 URL を本文に入れて即時入力を促す。将来 LINE 連携で解消。
- [図面セット判定がファイル数ベースのため誤判定の余地] → ステータスは参考表示と位置づけ、最終確認は人が行う運用を明記。
- [GAS の 6分/実行 制限] → 案件数×スロット数は高々数百行規模で問題ないが、巡回処理は案件単位で分割し途中経過を書き戻す実装とする。
- [警察署ごとに様式・運用差異がある可能性] → テンプレート Doc を `config` で案件ごとに差し替え可能にしておく。
- [期間短縮・中断時の扱い] → 自動削除しない設計（D3）のため、不要スロットは手動アーカイブ。運用手順書に記載。

## Migration Plan

新規導入のためデータ移行はなし。導入手順:

1. 組織アカウントで管理表・ルートフォルダ・GAS プロジェクトを作成し、`config` を設定。
2. `doc03.docx` を Docs 変換しテンプレート化、体裁をユーザーと確認（受入ポイント①）。
3. 進行中案件（現在窓口申請中の現場）を案件マスタに登録し、実データでスロット展開・Doc 生成を検証（受入ポイント②）。
4. アラートを管理者のみ宛てで1週間試運転 → 問題なければ通知先を本番宛先に切替（受入ポイント③）。

ロールバック: トリガーを無効化するだけで従来の手作業運用に戻れる（生成済み Doc はそのまま利用可）。

## Open Questions

- 期間欄の時刻の既定値（終日相当の表記）を警察署の受理実績に合わせてどう書くか — 1回目の監督作成分の実物を確認して決定する。
- 通知先: 案件ごとの担当者メールに加え、秘書課の共有メーリングリストを常時 CC するか。
- 「受領したが次回申請がまだ」の状態を独立アラートにするか（現状は受領アラートと同一トリガー内で種別分けする想定）。
